Généralités

Modèles de Fichiers

Afin d'étudier les méthodes d'accès aux fichiers dans un cadre général, on modélise la Mémoire Secondaire (MS) par une zone contiguë de blocs numérotés séquentiellement (ces numéros représentent les adresses de blocs).
NOTA : En mode graphique, on limite la taille de la MS à moins de dix (10) blocs.

Les blocs sont des zones contiguës d'octets de même taille, renfermant, entre autre, les données
(enregistrements) des fichiers.

Dans le schéma suivant, vous trouvez une MS de 10 blocs, dont 3 ne sont pas utilisés :

Guide N° 01 : Organisation globale des blocs

Dans les structures simples de fichiers, on a  deux possibilités distinctes d'organiser les blocs :
    1- soit le fichier est « vu comme un tableau » : tous les blocs qui le forment sont contigus.


2- soit le fichier est « vu comme une liste » : les blocs ne sont pas forcément contigus, mais sont chaînés entre eux.

Guide N° 02 : Organisation interne des blocs

    Les blocs sont censés contenir les enregistrements d'un fichier. Ces derniers peuvent être de longueur fixe ou variable : 

    1- Enregistrements de taille fixe : chaque bloc pourra alors contenir un  tableau d'enregistrements de même type.

    
    2- Enregistrements de taille Variable : chaque enregistrement dans le bloc sera vu comme étant une chaîne de cararactère (de longueur variable).

Guide N° 03 : Taxonomie des structures simples de fichiers

   En combinant entre l'organisation globale des fichiers (tableau ou liste) et celle interne aux blocs (format fixe ou variable des enregistrements), on peut définir une classe de méthodes d'accès (dites « simples ») pour organiser des données dans MS.
    Si de plus on prend en compte la possibilité de garder le fichier ordonné ou non, suivant les valeurs d'un champ clé particulier, on doublera le nombre de méthodes dans cette classe de structures simples de fichiers.

-Utilisons la notation suivante:
T : pour fichier vu comme tableau;     L : pour fichier vu comme liste.
O : pour fichier ordonné;     nO : pour fichier non ordonné.
F : pour format fixe des enregistrements;     V : pour format variable.
C : avec chevauchement des enregistrements entre blocs;     nC : sans chevauchement.

    Clé : La clé de l'enregistrement.

Guide N° 01 : Présentation de la structure TOF avec ZD (Partie 1)

La structure simple TOF avec ZD est un fichier constititué de :  
    1- Zone Primaire : un fichier ordonné TOF.
    2- Zone de Débordement (ZD)  : un fichier non ordonné LnOF.

Conseils : Consulter les parties TOF et LnOF pour plus de détails sur ces structures.

Caracactéristiques et Utilité d'un Fichier TOF avec ZD : 
    Pour gérer les insertions et afin d'éviter les décalages inter blocs, une zone de débordement est maintenue contenant les enregistrements expulsés des blocs de la zone primaire éventuellement lors des décalages intra blocs.
Pour pouvoir retrouver les enregistrements déplacés dans la zone de débordement, un lien entre les blocs de la zone primaire est utilisé comme tête de liste de blocs en débordement.

Recherche ( ? ) : 
La recherche est dichotomique dans la zone primaire suivit par une recherche séquentielle dans la zone de débordement.

Insertion ( + ) :

  • Dense

Guide N° 01 : Caractéristiques et utilité

   Avec les structures de fichiers simples, quand le fichier de données devient volumineux, les opérations d'accès (recherche, insertion, ...) deviennent très inefficaces. Les méthodes d'index permettent d'améliorer, dans une certaine mesure, les performances en gérant une structure auxiliaire (table d'index) accélérant la recherche.

    Un index est (généralement) une table ordonnée contenant les couples (Clé, Adr)  utilisée pour accélérer la recherche des enregistrements d'un fichier. Le champs 'Clé' représente une clé d'un enregistrement du fichier et le champs 'Adr' représente son adresse dans le fichier.

Guide N° 02 : L'index dense et l'index non dense

    On distingue deux (2) types d'index selon le type du fichier utilisé (Ordonné ou Non ordonné) :

    1- Index dense : il contient toutes les clés du fichier. Il est utilisé avec les fichiers non ordonnés.

    
    2- Index non dense : il contient uniquement les clés des derniers enregistrements de chaque bloc du fichier. Il est utilisé avec les fichiers ordonnés.

Guide N° 01 : Présentation générale

   Un arbre est une structure représentant un ensemble de valeurs organisées hiérarchiquement. Chaque valeur est stockée dans un noeud. Les noeuds sont connectés entre eux par des relations parentés.

   A part le noeud racine, tous les autres noeuds ont exactement un seul noeud parent et zéro ou plusieurs noeuds fils. Le noeud racine n'a pas de parent et possède zéro ou plusieurs fils.

Guide N° 02 : Les arbres en MS

    Les méthodes par tables d'index sont limités à certains type de fichiers (petits fichiers ou fichiers statiques). Les méthodes basées sur les structures d'arbres sont mieux adaptées aux fichier volumineux et/ou dynamiques.

   Afin de mieux occuper l'espace des blocs, on utilise des arbres m-aire où chaque noeud est représenté dans un bloc d'E/S.

Guide N° 01 : Présentation de l'arbre m-aire (Partie 1)

    Un arbre de recherche m-aire est la généralisation d'un arbre de recherche binaire. Un arbre de recherche m-aire d'ordre n est un arbre où chaque noeud peut avoir au maximum n fils et (n-1) valeurs.
Le degré (d) d'un noeud (le nombre de fils dans un noeud) est le nombre de valeurs stockées plus un.

Caractéristiques d'un bloc (noeud) :  
     TabVal : un tableau contenant les enregistrements du bloc.
     TabFils : un tableau contenant les lien vers les différents fils du noeud.
     Degré : un entier égale au nombre de fils du noeud (ou bien, au nombre d'enreg. moins un).

Caractéristiques d'un enregistrement :  
     Clé : le champ "Clé" de l'enregistrement.
     Bool : le bool de la suppression logique.

Guide N° 03 : Opérations de base sur l'arbre m-aire (Partie 1)

Recherche ( ? ) : 
La recherche d'une valeur C commence dans le noeud racine P et se poursuit le long d'une branche :
      1- Si C existe dans P alors la recherche s'arrête avec succès.
      2- Si C n'existe pas dans P alors :  soit k la position dans P où devrait être insérée C (pour que les valeurs restent ordonnées)
                   Si Fils N°k est différent de NULL alors P ← Fils N°k ; aller à 1
                   Sinon la recherche s'arrête avec échec.

L'insertion se fait dans le bloc retourné par la recherche. Elle provoque des décalages dans le bloc.
NOTA : Si on le bloc est plein, on alloue un nouveau bloc qui sera un fils du bloc plein.

Suppression ( - ): 
La suppression est logique (Mettre le champ "Bool" à vrai).

Fichiers avec Hachage

  • Hachage :

  • Statique
  • Dynamique
Guide N° 01 : Présentation générale

    Les techniques d'hachage sont bien utilisées pour l'accès au fichier se trouvant sur disque. L'objectif principal est de minimiser le nombre d'accès au fichier.
Le fichier est généralement divisé en N blocs contenant chacun b enregistrements. Une collision apparaît quand plus de b enregistrements ont la même adresse de bloc.

    L'Hachage consiste à stocker des données (x) dans une table (T) en utilisant une fonction (h) pour la localisation rapide (calcul d'adresse).
On essaye de stocker x dans la case d'indice h(x) (adresse primaire), si la case est déjà occupée(collision), on insère x a un autre emplacement (adr secondaire) déterminé par un algorithme donné (Méthode de résolution de collisions)

Méthodes de résolution de collisions : Lors de l'insertion de x, si l'adresse primaire h(x) est déjà utilisée par une autre donnée, la méthode de résolution de collision permet de trouver un autre emplacement (libre) pour x

Guide N° 02 : L'Hachage statique et l'hachage dynamique

On distigue deux (2) types différents d'hachage : 

   1- Hachage Statique : Ce type d'hachage est lié au nombre fixe de blocs (soit N blocs) que le fichier contient. D'une autre manière, la fonction d'hachage h(x) est toujours stable (elle renvoie des valeurs entre 0 et (N - 1) ).
Si le nombre d'insertion devient important, les performances se dégradent à cause des nombreuses données en débordement.

 2- Hachage Dynamique : Ce type d'hachage est lié au nombre variable de blocs. D'une autre manière, la fonction    d'hachage h(x) change en fonction du nombre de blocs, et ceci permet de garder les performances .

Guide N° 03 : Opérations de base (Partie 1)

Recherche ( ? ) : 
La recherche d'une clé x consiste tout d'abord à calculer h(x)  pour récupérer le numéro du bloc primaire. Si la clé n'existe pas dans le bloc primaire, on recherche séquentiellement dans la zone de débordement (si elle existe).

Insertion ( + ) : 
L'insertion se fait dans le bloc primaire retourné par la fonction h(x). Si le bloc primaire est plein (collision), on inserère dans la zone de débordement.

Suppression ( - ): 
La suppression se fait par écrasement de l'enregistrement à supprimer par le dernier enregistrement de la liste associée au bloc primaire.

Guide N° 01 : Hachage dynamique "hachage Linéaire" (Partie 1)

   Dans les méthodes du hachage dynamique, la fonction de hachage h change dynamiquement pour s'adapter a la taille du fichier, ce qui permet de garder de bonnes performances même si le nombre d'insertion (ou de suppression) augmente.


Hachage linéaire :  c’est l'une des méthodes de l’hachage dynamique parmi les plus performantes. Le fichier est formé par une suite de blocs (qu'on appelle ≪ cases ≫).
On utilise 2 fonctions de hachage :
hi(x) = x mod 2 puiss i et hi+1(x) = x mod 2 puiss (i+1)

Le nombre de cases augmente et diminue en fonction des insertions  et des Suppressions.

Guide N° 01 : Présentation générale

    Les systèmes de bases de données utilisent des langages de requêtes de haut niveau pour la consultation de grands volumes de données.

L’exécution de ce type de requêtes passe par leurs traduction en opérations de base (scan, tri, jointure…). Il existe un ou plusieurs algorithmes pour chacune de ces opérations de base.

L'implémentation efficace de ces opérations de base dépend souvent de la taille de la zone mémoire
de travail allouée (en MC).

Guide N° 02 : La fusion et la jointure

Nous voyons dans ce chapitre l’implémentation de quelques opérations de base utilisées dans le
traitement des requêtes de haut niveau :

- Fusion : C'est une opération qui a comme but de concaténer deux fichiers ordonnés en gardant l'ordre dans le fichier résultat.

- Jointure : Cette opération consiste à rassembler toutes les correspondances dans deux fichiers ordonnés dans in nouveau fichier.

Guide N° 01 : Fusion des fichiers ordonnés (Partie 1)

   La fusion est une opération qui consiste à concaténer deux fichiers ordonnés. Le fichier résultat gardre aussi l'orde des clés.

Caractéristiques et besoins :
Afin de réaliser cette opération, il faut : 
    1- Réserver trois (3) emplacements dans la mémoire secondaire (MS).
    2- Utiliser trois (3) buffers dans la mémoire centrale (MC).

Coùt de l'opération :
La réalisation de cette opération a besoin de parcourir les deux fichiers  en passant par tous les blocs et leurs enregistrements (i.e. elle couteuse).

Guide N° 01 : Jointure des fichiers ordonnés (Partie 1)

   La jointure de deux fichiers est un genre de produit cartésien avec un filtre, sous forme de condition permettant de sélectionner les enregistrements à concaténer dans le résultat.

Soient F1 et F2 deux fichiers de tailles respectives N1 et N2 blocs. La jointure de F1 et F2 selon une condition C, consiste pour chaque couple d’enregistrements de F1 et F2, à produire un enregistrement en concaténant e1 et e2 dans un nouveau fichier résultat, à condition que les attributs de e1 et e2 vérifient C.
NOTA : La condition utilisée est l'égalité ( = ).

Coùt de l'opération : Jointure par "boucles imbriquées"
La réalisation de cette opération a besoin de parcourir pour chaque bloc de F1, tous les blocs de F2 (i.e. elle très couteuse).
